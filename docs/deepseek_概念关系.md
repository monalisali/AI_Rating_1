# 税务AI系统开发文档：动态提示词生成与推理优化

>1. 主要解决：同一个问题且获取到所有文章后，AI回答不稳定的问题 
>2. 本文档面向**开发人员**和**Claude Code（AI代码生成）**，描述如何通过动态生成四个中间产物（条件-结论对、政策场景枚举、概念关系断言、时间约束），解决模型归纳错误、遗漏和概念混淆问题。所有提示词可直接使用。


---

## 1. 概述

### 1.1 问题背景
税务AI系统工作流程：用户提问 → 检索知识库文章 → 模型推理回答。存在三大问题：
- **遗漏**：模型未覆盖用户问题涉及的所有政策场景（如西部大开发优惠）。
- **归纳错误**：模型自由归纳时错误组合条件或乱叠加以至于结论错误。
- **概念混淆**：模型将不同层次的概念等同（如“两免三减半”等同于“减半期计算规则”），导致推理链条断裂。

### 1.2 解决方案核心思路
不依赖模型自由推理，而是**先让系统自动生成四个结构化产物**，再将这些产物作为硬约束注入最终提示词，强制模型按填空式逻辑回答。

四个产物（Skill）：
1. **条件-结论对**：将文章拆解为`(条件, 结论, 来源ID)`原子单元。
2. **政策场景枚举**：列出用户问题可能涉及的所有政策场景标签（如“西部大开发”“小微企业”）。
3. **概念关系断言**：标注概念之间的逻辑关系（上下位、同义、相关但不同义、互斥、继承、属性）。
4. **时间约束**：提取政策的生效、失效、过渡期条件。

### 1.3 整体数据流
```
用户问题 + 检索文章列表
    │
    ├─► Skill 1: 条件-结论对抽取 ──► 产物A
    ├─► Skill 2: 政策场景枚举 ──► 产物B
    ├─► Skill 3: 概念关系断言 ──► 产物C → 规则校验层 → 产物C'
    └─► Skill 4: 时间约束抽取 ──► 产物D
    │
    └─► 动态组装最终提示词 → 模型生成答案
```

所有Skill可**并行调用**，最后用代码合并产物。

---

## 2. 整体流程（实现级描述）

**输入**：
- `user_query`: 字符串，用户原始问题。
- `articles`: 列表，每个元素包含 `id`（文章唯一标识）、`title`、`content`、`source`（法规文号可选）、`publish_date`（可选）。

**输出**：
- `final_answer`: 字符串，模型最终回答。

**步骤**：
1. 并行调用四个Skill（各自独立HTTP请求），得到产物A、B、C、D。
2. 对产物C（概念关系断言）执行规则校验层（纯Python代码），输出清洗后的`C_cleaned`。
3. 将所有产物格式化为文本块。
4. 使用**最终回答提示词模板**，将`user_query`和上述文本块填充，调用LLM生成答案。
5. 返回答案。

> 所有LLM调用均设置 `temperature=0`（若API不允许0则用0.01），保证确定性输出。

## 3. 四个动态产物的详细定义

### 3.1 Skill 1: 条件-结论对抽取

**目标**：将每篇文章拆分为若干个`(条件, 结论, 来源文章ID)`，每个条件-结论对表达一条独立规则。

**输入**：
- `article_content`: 字符串，单篇文章的正文。
- `article_id`: 字符串。

**输出**：JSON数组，每个元素：
```json
{
  "condition": "企业属于小微企业且应纳税所得额≤100万",
  "conclusion": "减按12.5%计入应纳税所得额",
  "article_id": "ART001",
  "evidence": "原文摘录（可选）"
}
```

**提示词（System Prompt）**：
```text
你是一个税务规则抽取专家。请从给定的文章中，抽取所有明确的“条件→结论”对。

输出格式：JSON数组，每个对象必须包含：
- condition: 触发条件（一句话，客观可判断）
- conclusion: 对应的结论/规则
- article_id: 输入的文章ID（原样输出）

要求：
1. 每条规则独立，不要合并多个条件。
2. 如果文章包含例外或限制条件，需单独拆分为带否定的条件。
3. 不要添加原文没有的信息。
4. 如果文章无明确规则，输出空数组[]。
5. 只输出JSON，不要其他解释。
```

**用户提示词示例**：
```text
文章ID: VAT_2025_001
文章内容:
"自2025年1月1日起，小规模纳税人发生增值税应税销售行为，合计月销售额未超过10万元的，免征增值税。月销售额超过10万元但未超过30万元的，减按1%征收增值税。"
```

**模型输出示例**：
```json
[
  {
    "condition": "小规模纳税人，月销售额未超过10万元",
    "conclusion": "免征增值税",
    "article_id": "VAT_2025_001"
  },
  {
    "condition": "小规模纳税人，月销售额超过10万元但未超过30万元",
    "conclusion": "减按1%征收增值税",
    "article_id": "VAT_2025_001"
  }
]
```

### 3.2 Skill 2: 政策场景枚举

**目标**：从用户问题和所有检索文章中，提取所有可能适用的政策场景标签（去重），用于强制模型逐项检查。

**输入**：
- `user_query`: 字符串
- `articles`: 列表，每个包含`title`, `content`（不限制数量，可拼接后输入）

**输出**：JSON数组，每个元素为场景标签字符串。
```json
["小微企业优惠", "高新技术企业优惠", "西部大开发优惠", "海南自贸港优惠"]
```

**提示词（System Prompt）**：
```text
你是一个税务场景分类专家。给定用户问题和相关文章，请提取所有可能适用的政策场景标签。

输出格式：JSON数组，每个元素是一个简短的场景标签（如“小微企业优惠”）。

要求：
1. 标签来源于文章中的政策名称或常见税务术语。
2. 标签应具有区分度，避免过于宽泛（如“税收优惠”不可取，应细化为“西部大开发优惠”）。
3. 不要遗漏文章中明确提到的场景。
4. 只输出JSON数组，不要其他解释。
```

**用户提示词示例**：
```text
用户问题：现行企业所得税法下，居民企业什么情况下取得的所得可能适用7.5%税率？

相关文章：
[文章1] 标题：海南自由贸易港鼓励类产业企业减按15%税率征收企业所得税
内容：对注册在海南自贸港并实质性运营的鼓励类产业企业，减按15%的税率征收企业所得税。其中，对旅游业、现代服务业、高新技术产业企业，2025年前新增境外直接投资取得的所得，免征企业所得税...
[文章2] 标题：财政部 税务总局关于海南自由贸易港企业所得税优惠政策的通知
内容：...符合条件的企业，可按规定享受减按15%税率。此外，对在海南自贸港设立的旅游业、现代服务业、高新技术产业企业，其2025年前新增境外直接投资取得的所得，免征企业所得税。
```

**模型输出示例**：
```json
["海南自贸港鼓励类产业优惠", "旅游业/现代服务业/高新技术产业境外投资所得免税"]
```

### 3.3 Skill 3: 概念关系断言抽取

**目标**：识别文章中出现的概念对及其逻辑关系，用于防止模型混淆。

**输入**：同Skill 2（用户问题 + 所有文章内容）

**输出**：JSON数组，每个元素：
```json
{
  "entity_a": "两免三减半",
  "entity_b": "定期减免税的减半期计算规则",
  "relation_type": "related_not_equal",
  "evidence": "前者是完整的定期减免政策，后者是计算规则，不可等同"
}
```
关系类型枚举：`hypernym`, `synonym`, `related_not_equal`, `mutually_exclusive`, `succession`, `property_of`。

**提示词（System Prompt）**：
```text
你是一个税务概念关系抽取专家。请从给定的用户问题和文章中，抽取所有成对概念之间的逻辑关系。

关系类型严格限定为以下六种（必须从列表中选）：
- hypernym: A是B的一种（下位→上位）
- synonym: A和B完全等价
- related_not_equal: A和B相关但不可等同（警告模型不要划等号）
- mutually_exclusive: A和B不能同时适用
- succession: B替代A（可能带过渡期）
- property_of: A是B的一个属性/参数

要求：
1. 每条断言必须提供原文证据（evidence字段，摘录原文1-2句）。
2. 只抽取文章或问题中明确体现的关系，不要臆想。
3. 输出JSON数组，即使没有关系也输出空数组[]。
4. 不要输出其他解释。
```

**用户提示词示例**：
```text
用户问题：西部大开发企业能否享受两免三减半？

相关文章：
[文章1] 标题：财税〔2001〕202号（已废止）...（内容略）
[文章2] 标题：财税〔2011〕58号 关于深入实施西部大开发战略有关税收政策问题的通知
内容节选：五、本通知自2011年1月1日起执行。《财政部 国家税务总局关于西部大开发税收优惠政策适用目录变更问题的通知》（财税〔2001〕202号）......同时停止执行。
三、对西部地区2010年12月31日前新办的、根据《财政部 国家税务总局关于西部大开发税收优惠政策适用目录变更问题的通知》（财税〔2001〕202号）可以享受“两免三减半”优惠的企业，其优惠可以继续享受到期满为止。
```

**模型输出示例**：
```json
[
  {
    "entity_a": "两免三减半",
    "entity_b": "定期减免税",
    "relation_type": "hypernym",
    "evidence": "两免三减半是指第一年至第二年免征，第三年至第五年减半征收，属于定期减免税的一种形式。"
  },
  {
    "entity_a": "财税〔2001〕202号",
    "entity_b": "财税〔2011〕58号",
    "relation_type": "succession",
    "evidence": "财税〔2011〕58号第五条规定财税〔2001〕202号停止执行，但第三条规定存量企业过渡期保留。"
  }
]
```

### 3.4 Skill 4: 时间约束抽取

**目标**：提取政策的时间适用范围（生效、失效、过渡期条件）。

**输入**：同Skill 2

**输出**：JSON数组，每个元素：
```json
{
  "policy_name": "财税〔2001〕202号两免三减半",
  "constraint_type": "valid_for|invalid_for|transitional",
  "condition": "2010年12月31日前新办企业",
  "article_id": "ART002"
}
```

**提示词（System Prompt）**：
```text
你是一个税务时间信息抽取专家。从文章中提取所有与政策适用时间相关的约束。

输出格式：JSON数组，每个对象包含：
- policy_name: 政策名称或简称
- constraint_type: 枚举值 "valid_for"（适用条件）, "invalid_for"（不适用条件）, "transitional"（过渡期保留）
- condition: 具体的条件描述（如“2010年12月31日前新办企业”）
- article_id: 来源文章ID

要求：
1. 只抽取明确的时间限定，如“自X年X月X日起执行”“停止执行”“继续享受到期满”。
2. 如果政策有废止但保留存量，拆分为两条：一条invalid_for（对新办企业），一条transitional（对存量）。
3. 输出空数组若无时间信息。
```

**用户提示词示例**：同上文类似，展示文章2内容。

**模型输出示例**：
```json
[
  {
    "policy_name": "财税〔2001〕202号两免三减半",
    "constraint_type": "invalid_for",
    "condition": "2011年1月1日以后新办企业",
    "article_id": "ART002"
  },
  {
    "policy_name": "财税〔2001〕202号两免三减半",
    "constraint_type": "transitional",
```

## 4. 动态组装最终提示词

假设我们已经获得以下变量：
- `user_query`: 原始问题
- `condition_conclusion_pairs`: 产物A的JSON字符串
- `scene_enum`: 产物B的JSON字符串
- `concept_constraints`: 产物C经过校验后转换成的自然语言约束列表（见下文转换）
- `time_constraints`: 产物D的JSON字符串

### 4.1 概念约束转换

将产物C（断言数组）转换为自然语言约束语句，不需要LLM，可通过模板生成。例如：

```python
constraint_texts = []
for a in cleaned_assertions:
    if a['relation_type'] == 'hypernym':
        text = f"“{a['entity_a']}”是“{a['entity_b']}”的下位概念。禁止将二者视为并列或等同。"
    elif a['relation_type'] == 'synonym':
        text = f"“{a['entity_a']}”与“{a['entity_b']}”为同义概念，回答时可直接等同替换。"
    elif a['relation_type'] == 'related_not_equal':
        text = f"“{a['entity_a']}”与“{a['entity_b']}”相关但不可等同。禁止在推理中将二者相互替换或视为相同。"
    elif a['relation_type'] == 'mutually_exclusive':
        text = f"“{a['entity_a']}”与“{a['entity_b']}”互斥，不能同时适用。"
    elif a['relation_type'] == 'succession':
        text = f"“{a['entity_b']}”替代“{a['entity_a']}”，但注意是否存在过渡期保留（见时间约束）。"
    elif a['relation_type'] == 'property_of':
        text = f"“{a['entity_a']}”是“{a['entity_b']}”的一个属性/参数，不等同于整体政策。"
    constraint_texts.append(text)
```

### 4.2 最终回答提示词模板

```text
你是一个税务问答专家。请严格按以下结构和约束回答问题。

## 用户问题
{user_query}

## 必须检查的政策场景（不要遗漏任何一项）
{scene_enum_list}

## 逻辑约束（必须遵守）
{每条约束语句换行}

## 时间适用性约束
{time_constraints_formatted}

## 可用的条件-结论对（推理依据）
{condition_conclusion_pairs_formatted}

## 回答步骤
1. 对【必须检查的政策场景】中的每一个场景，判断是否适用，并引用【可用的条件-结论对】中的对应条目。
2. 检查政策之间是否存在互斥或叠加限制，依据【逻辑约束】。
3. 最终输出必须：
   - 列出所有适用的场景及其结论。
   - 如果有多种适用情况，明确说明它们是否可以同时享受，若不能则指出优先级。
   - 给出最终的汇总答案。

请现在开始回答。
```

### 4.3 填充示例

假设：
- `user_query = "现行企业所得税法下，居民企业什么情况下取得的所得可能适用7.5%税率？"`
- `scene_enum_list` = `["海南自贸港鼓励类产业优惠", "西部大开发优惠", "小型微利企业优惠"]`
- `concept_constraints` = `["“海南自贸港15%税率”与“境外投资所得免税”相关但不可等同"]`
- `time_constraints` = 无
- `condition_conclusion_pairs` 包含相应条款。

最终提示词即填充上述模板后发送给LLM（temperature=0.3～0.5，最终回答可略高）。

## 5. 最终回答生成

调用同一个LLM（或同一API），使用上一节的模板。注意：
- 温度设为0.3以避免过于死板，但不宜超过0.5。
- 最大输出token根据情况设为2000。

输出直接作为最终答案返回给用户。

## 6. 错误处理与降级策略

1. **任一Skill超时或返回格式错误**：跳过该产物，但其他产物继续。例如概念关系失败则不在最终提示词中包含约束，仅使用条件-结论和场景枚举。
2. **规则校验层发现大量冲突**：丢弃整个断言数组，记录日志，后续可人工分析。
3. **最终回答生成失败**：返回兜底消息“系统暂无法回答此问题，请稍后重试或咨询税务专员。”

| ID | 规则 | 实现逻辑 |
|----|------|----------|
| C1 | 删除自反关系 | `if a['entity_a'] == a['entity_b']: skip` |
| C2 | 去重 | 使用`(e1,e2,rel)`元组作为key，合并evidence |
| C3 | 同义+互斥冲突 | 若同一对同时存在synonym和mutually_exclusive，删除mutually_exclusive |
| C4 | 同义+上下位冲突 | 若同时存在，删除hypernym |
| C5 | 补充互斥对称 | 若存在(A,B,mutually_exclusive)且无(B,A)，则添加，标记derived |
| C6 | 证据实体匹配 | `if a['entity_a'] not in evidence or a['entity_b'] not in evidence: delete` |
| C7 | 证据长度限制 | 若evidence长度>500字符，截断至200 |
| C8 | 单源降级 | 若断言只来自一篇文章，添加`confidence:"medium"`；多源则`high` |
| C9 | 时间顺序校验（若含时间字段） | 若succession且旧政策废止年份晚于新政策生效年份 → 改为`related_not_equal` |

建议实现C1～C7作为第一阶段。
